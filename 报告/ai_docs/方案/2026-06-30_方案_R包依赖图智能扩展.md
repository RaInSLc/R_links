# R包依赖图智能扩展方案

> 生成日期: 2026-06-30 | 类型: 方案 | 状态: 草稿

## 背景

当前项目已经具备 R 包输入解析、多源检索、缓存、脚本生成和前端报告展示能力。现有检索结果主要回答“输入包在哪里、版本是什么、应该用什么来源安装”，但还不能回答“这些包之间有什么依赖关系、哪些依赖是重依赖、重依赖本身还会带来哪些包”。

用户期望输入 3 个包时，系统不仅显示这 3 个包之间的关系，还能继续拉取它们的外部依赖，并按重度、轻度等关系强度展示依赖链路。该能力本质上不是单纯的搜索增强，而是从“包来源验证”升级到“依赖图分析”。

## 问题分析

### 现状

1. 后端 `search_packages` 以输入包为单位并发检索 CRAN、Bioconductor 和 GitHub，返回 `SearchResponse { run_id, results, logs, stopped }`。
2. 当前 `SearchResult` 只包含包名、版本、仓库、来源、状态和提示信息，没有依赖字段。
3. 前端 `useSearch` 维护的是扁平 `SearchResult[]`，`ReportView` 也以表格方式展示“来源验证”结果。
4. 脚本生成当前依赖 `SearchResult[]` 和安装策略开关，重点是生成安装命令，而不是构建依赖闭包。

### 根因

现有模型是“输入包 -> 多来源结果”的扁平结构，缺少表达以下关系的数据层：

1. 包与包之间的有向边。
2. 依赖关系类型，例如 `Depends`、`Imports`、`LinkingTo`、`Suggests`、`Enhances`。
3. 依赖强度和层级深度。
4. 多个根包共享依赖、循环依赖、依赖缺失、依赖来源不确定等图分析状态。

因此不建议把依赖信息直接塞进现有 `SearchResult.message` 或扩展成字符串字段，否则后续 UI、缓存、脚本生成都会变得难维护。

## 核心发现

1. 该功能不需要推倒重写，但需要一次中等规模的模型扩展。
2. 最合理的架构是保留现有来源检索链路，再新增独立的依赖解析模块，最终让 `SearchResponse` 携带可选 `dependencyGraph`。
3. 依赖强度应基于 R 包 `DESCRIPTION` 字段和图结构共同判断：`Depends`、`Imports`、`LinkingTo` 属于默认重依赖，`Suggests`、`Enhances` 属于轻依赖；被多个输入包共同引用、或本身继续引入大量重依赖的包，可在 UI 中标记为高影响依赖。
4. 第一阶段应优先做“可视化和解释”，不要急着改变安装脚本。R 自身安装器已经能处理大多数依赖，先把依赖关系展示清楚更稳。
5. 必须设置深度、节点数、请求数和缓存上限，否则输入少量包也可能扩展出大量网络请求。

## 实施方案

### 方案概述

新增一个后端依赖解析阶段：输入包完成来源验证后，后端根据命中的真实包名和来源拉取元数据，解析 `DESCRIPTION` 依赖字段，构建依赖图，并返回给前端展示。前端在检索报告中新增“依赖图 / 依赖清单”视图，展示根包、直接依赖、重依赖、轻依赖、共享依赖和重依赖的下一层依赖。

推荐的后端数据模型如下：

```rust
pub struct DependencyGraph {
    pub roots: Vec<String>,
    pub nodes: Vec<DependencyNode>,
    pub edges: Vec<DependencyEdge>,
    pub summary: DependencySummary,
}

pub struct DependencyNode {
    pub package: String,
    pub source: String,
    pub version: String,
    pub depth: usize,
    pub root_packages: Vec<String>,
    pub direct_dependency_count: usize,
    pub heavy_dependency_count: usize,
    pub status: String,
}

pub struct DependencyEdge {
    pub from: String,
    pub to: String,
    pub relation: String,
    pub strength: String,
    pub depth: usize,
}
```

推荐的依赖强度规则如下：

| DESCRIPTION 字段 | 默认强度 | 展示含义 |
|------------------|----------|----------|
| Depends | 重度 | 加载或运行高度相关 |
| Imports | 重度 | 运行时调用依赖 |
| LinkingTo | 重度 | 编译期依赖 |
| Suggests | 轻度 | 测试、示例、可选功能依赖 |
| Enhances | 轻度 | 反向增强关系，默认不展开 |

### 预期效果

1. 输入多个包后，报告能显示哪些包互相依赖。
2. 能区分重依赖和轻依赖，并解释依赖来源字段。
3. 重依赖可继续展开其自身依赖，用户能看到隐藏的依赖包。
4. 共享依赖可被识别，例如多个输入包共同依赖同一个基础包时，在 UI 中标记为“共享依赖”。
5. 现有安装命令生成逻辑可以保持稳定，后续再选择是否基于依赖图生成更明确的安装顺序。

## 最终建议

建议按“中等规模扩展”实施，不需要大重构。核心做法是新增依赖图模型和依赖解析模块，现有 `search_packages` 保持为主流程，只在主包检索完成后追加依赖分析阶段。

优先级建议如下：

1. MVP：只解析 CRAN 和 Bioconductor 的直接依赖，展示重依赖、轻依赖和共享依赖。
2. 第二阶段：对重依赖展开一层，支持 GitHub `DESCRIPTION` 解析。
3. 第三阶段：增加图形化视图、依赖缓存管理、安装脚本顺序优化。

重构范围判断：

1. 后端需要新增模块和模型，属于中等改动。
2. 前端需要扩展报告页，属于中等改动。
3. 脚本生成可以暂不改，属于低改动。
4. 现有搜索逻辑不需要推倒重写，只需要在响应模型和结束阶段挂接依赖图。
