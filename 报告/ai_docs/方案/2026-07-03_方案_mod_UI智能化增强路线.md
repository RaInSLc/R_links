# mod_UI 智能化增强路线方案

> 生成日期: 2026-07-03 | 类型: 方案 | 状态: 已确认

## 背景

当前 `mod_UI` 已具备 R 包安装命令生成、多源检索、缓存、历史记录、依赖图、镜像测速、反向依赖分析和安装后验证等能力。用户反馈指出当前软件“联想太差、不够聪明”，说明问题重点不再是基础功能缺失，而是软件对输入意图、包来源、安装环境、失败原因和下一步操作的自动推断能力不足。本方案用于规划后续智能化增强路线。

## 问题分析

### 现状

现有能力偏向“用户输入明确指令后执行”：用户需要知道包名、来源、安装方式、是否检索、是否验证、是否使用缓存等。系统虽然能多源检索，但对混乱输入、错误拼写、上下文历史、系统环境、检索失败、依赖复杂度和安装策略的主动推荐不足。

### 根因

当前逻辑主要基于规则和显式开关，缺少“意图识别层”“候选推荐层”“置信度解释层”和“失败恢复层”。因此软件能生成命令，但不会充分解释为什么选这个命令、还有哪些更合适的选择、当前输入里哪些内容可能不是包名、失败后应该先尝试哪个替代路径。

## 核心发现

1. 输入理解需要增强：应能识别从论文、GitHub README、R 脚本、报错日志、安装命令中粘贴的混合文本，并自动提取包名、GitHub 仓库、URL、版本号和来源提示。
2. 包名联想需要增强：应支持拼写纠错、大小写纠正、常见别名映射、Bioconductor/CRAN/GitHub 同名包消歧。
3. 策略推荐需要增强：应根据检索结果、版本要求、包来源、系统环境和依赖复杂度，自动推荐 `install.packages`、`BiocManager::install`、`remotes::install_github`、`remotes::install_version` 或归档 URL 安装。
4. 失败恢复需要增强：检索失败、GitHub 限流、CRAN 包下架、依赖解析失败时，应给出可执行的下一步，而不是只显示失败状态。
5. 历史与缓存需要变成“记忆”：系统应基于历史成功命令、缓存命中、用户常用镜像和常用来源，优先推荐更符合用户习惯的命令。
6. 依赖图需要更智能：不仅展示依赖，还应识别重依赖、共享依赖、可能编译失败的系统依赖和安装顺序建议。
7. 交互应从“按钮工具”升级为“任务助手”：每一步给出建议动作，例如“检测到 GitHub 仓库，建议切换 GitHub 安装”“检测到版本号，建议安装指定版本”。

## 实施方案

### 方案概述

建议新增一个“智能建议层”，不直接替代现有规则，而是作为输入解析、检索结果和历史数据之上的解释与推荐模块。该层输出结构化建议，包括建议类型、置信度、影响范围、推荐操作和可一键应用的状态变更。前端以轻量建议卡片展示，后端负责核心推断和可复用规则。

| 步骤 | 操作 | 负责模块 | 风险等级 |

| 2 | 在工作台输入框下方展示“智能识别结果”：有效包、疑似噪音、版本要求、来源提示 | `src\WorkspaceView.tsx` | 中 |

| 4 | 增加策略推荐器：按来源、版本、归档状态、依赖复杂度生成推荐安装方式和理由 | `logic.rs` | 中 |

### 预期效果

完成后，用户粘贴混乱文本时，软件可以主动识别有效内容并解释过滤原因；检索后不仅展示结果，还能给出推荐命令、推荐理由和失败恢复路径。用户不需要理解所有安装工具差异，也能获得更合适的安装脚本。

## 风险评估

| 风险项 | 影响范围 | 可能性 | 缓解措施 |
|--------|----------|--------|----------|
| 智能推荐误判 | 安装命令生成 | 中 | 所有推荐必须可解释，并保留用户手动切换 |
| 输入识别过度过滤 | 包列表解析 | 中 | 将过滤结果可视化，支持一键恢复被忽略内容 |
| 历史偏好影响新任务 | 推荐排序 | 低 | 历史偏好只影响排序，不覆盖明确输入 |
| 交互复杂度上升 | 工作台 UI | 中 | 使用折叠式建议卡片，默认只展示最高优先级建议 |

## 最终建议

优先落地三个高收益能力：输入意图分析、策略推荐器、失败恢复建议。三者能直接改善“联想差、不聪明”的主观体验，并且可以在不大改现有安装命令生成核心的前提下逐步接入。第二阶段再做包名候选联想、历史偏好记忆和依赖图智能分析。
