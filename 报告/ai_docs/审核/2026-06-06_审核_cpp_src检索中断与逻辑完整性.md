# cpp_src 检索中断与逻辑完整性审核

> 生成日期: 2026-06-06 | 类型: 审核 | 状态: 已确认

## 背景

本次审核针对 `cpp_src\` 中的 C++ 原生桌面程序，重点检查批量检索是否存在逻辑错误、检索链路中断、无法停止、错误结果被当作有效结果、线程或网络句柄竞争等问题。

审核范围包括：

- `cpp_src\app_window.cpp`
- `cpp_src\app_window.hpp`
- `cpp_src\pkg_logic.cpp`
- `cpp_src\pkg_logic.hpp`
- `cpp_src\main.cpp`
- `cpp_src\build.bat`
- `cpp_src\build_test.bat`

本次结论属于可靠性与正确性审核。未发现可直接认定的安全漏洞；下文所述问题均按检索可用性、结果可信度和进程稳定性评估。

## 问题分析

### 现状

程序使用一个后台 `std::thread` 串行访问 CRAN、Bioconductor、r-universe 和 GitHub。UI 线程通过原子变量保存当前 WinINet 会话句柄与请求句柄，并在用户点击停止或关闭窗口时直接调用 `InternetCloseHandle`。

正常路径能够编译并完成基本检索，但停止、异常和网络失败路径缺少统一状态模型。当前实现只有“找到/未找到”两个业务结果，没有“已取消、网络失败、解析失败、限流、服务端错误”等独立状态。

### 根因

1. WinINet 句柄同时由 UI 线程和工作线程拥有，原子变量仅保护句柄值的读写，没有建立唯一关闭权。
2. 工作线程无论正常结束还是被停止，最终都投递同一个 `WM_SEARCH_COMPLETE`。
3. 网络层失败未向上返回结构化错误，最终统一落入“所有渠道均未找到”。
4. 异常清理、完成通知和业务结果生成分散在多个分支，没有统一的任务上下文和退出守卫。
5. Bioconductor 历史版本列表与解析规则采用硬编码，未随上游版本演进自动更新。

## 核心发现

### 1. 停止操作存在同步 WinINet 调用期间跨线程关闭句柄的问题

**影响等级：高**

**位置：**

- `cpp_src\app_window.cpp:229-242`
- `cpp_src\app_window.cpp:301-310`
- `cpp_src\app_window.cpp:652-656`
- `cpp_src\app_window.cpp:754-766`
- `cpp_src\app_window.cpp:1021-1029`

UI 线程在停止或销毁窗口时，对 `m_hActiveRequest` 和 `m_hInternet` 执行 `exchange(NULL)` 后立即关闭句柄。此时工作线程可能仍在同步执行 `InternetOpenUrlW`、`HttpQueryInfoW` 或 `InternetReadFile`。

微软文档明确说明，只有在没有 API 正在或将要使用该句柄时，关闭操作才是安全的。当前代码使用的是同步 WinINet 调用，不是返回 `ERROR_IO_PENDING` 的异步取消模型，因此当前跨线程关闭方式不满足该约束。

此外，`SafeRequestHandle` 在 UI 线程关闭请求句柄后仍保留原始 `m_hReq`。其析构或显式 `Close()` 会再次调用 `InternetCloseHandle(m_hReq)`。`compare_exchange_strong` 只尝试清空原子变量，不会撤销本地句柄的关闭动作。微软文档同时指出 HINTERNET 句柄值会被快速复用，因此重复关闭不仅是无效调用，极端情况下还可能影响后来复用相同数值的新句柄。

**结论：**该问题存在明确的检索异常终止、无效句柄访问、进程不稳定或后续请求异常路径，是当前最需要优先处理的问题。

### 2. 手动停止仍被报告为“全量体检结束”，并基于部分结果生成命令

**影响等级：高**

**位置：**

- `cpp_src\app_window.cpp:255-257`
- `cpp_src\app_window.cpp:275-276`
- `cpp_src\app_window.cpp:661`
- `cpp_src\app_window.cpp:709-721`

工作线程退出循环后无条件投递 `WM_SEARCH_COMPLETE`。UI 收到消息后无条件显示“全量体检结束”，统计当前已收到的部分结果，切换到 Auto 模式并重新生成安装命令。

当用户在第 N 个包检索期间停止时，后续包没有任何 `SearchProgressInfo` 记录。命令生成逻辑会把这些包当作“尚未检查”，并按默认 Auto/Base 路径生成命令，界面却提示全量完成。

**结论：**停止操作会产生“任务已完成”的错误状态，并可能输出基于不完整检索结果的安装命令。该问题不会形成无限循环，但会使中断后的结果不可信。

### 3. 网络故障、超时和服务端错误会被误判为“所有渠道均未找到”

**影响等级：高**

**位置：**

- `cpp_src\app_window.cpp:337-358`
- `cpp_src\app_window.cpp:365-466`
- `cpp_src\app_window.cpp:476-549`
- `cpp_src\app_window.cpp:556-638`
- `cpp_src\app_window.cpp:642-646`

除 GitHub 403 外，代码没有记录 `InternetOpenUrlW` 失败原因，也没有处理非 200 状态码、超时、TLS 错误、代理错误、DNS 错误、429 或 5xx。任一请求失败只会静默进入下一个源；全部源失败后统一投递“未找到”。

因此，断网、代理配置错误、证书错误或上游临时故障都可能让一个真实存在的包被标记为“不存在”。这属于确定性错误分类问题，不是单纯的日志不足。

**结论：**当前程序无法区分“包不存在”和“检索服务不可用”，会造成批量检索结果整体失真。

### 4. InternetOpenW 失败时没有完成通知，形成静默中断

**影响等级：高**

**位置：**

- `cpp_src\app_window.cpp:252-264`

当 `InternetOpenW` 返回空句柄时，工作线程只设置 `m_searchActive = false` 并直接返回，不投递日志、错误消息或 `WM_SEARCH_COMPLETE`。报告页面会停留在“开始扫描”状态，用户无法从 UI 判断任务已经退出。

**结论：**这是明确的“检索已经中断但界面无完成或失败反馈”路径。

### 5. 正常收尾与停止操作之间仍存在会话句柄重复关闭竞态

**影响等级：高**

**位置：**

- `cpp_src\app_window.cpp:652-656`
- `cpp_src\app_window.cpp:1021-1029`

工作线程先读取 `m_hInternet` 到 `hToClose`，再把原子变量设为空并关闭本地句柄。该过程不是单个原子交换。UI 线程可在“读取”和“置空”之间执行 `exchange(NULL)` 并关闭同一句柄，随后工作线程再次关闭。

**结论：**`std::atomic<HINTERNET>` 没有消除双重关闭；正常完成临界点点击停止仍可触发竞态。

### 6. 异常处理会吞掉错误，并可能遗留根会话句柄

**影响等级：中**

**位置：**

- `cpp_src\app_window.cpp:252-266`
- `cpp_src\app_window.cpp:652-661`

外层 `catch (...)` 为空。若在正常清理前抛出异常，局部 `hInt` 不受 RAII 管理，`m_hInternet` 可能保持旧值，错误原因也不会显示。随后仍投递完成消息，造成“异常结束被当作正常完成”。

### 7. Bioconductor 历史版本列表已经落后于当前正式版本

**影响等级：中**

**位置：**

- `cpp_src\app_window.cpp:420-424`
- `cpp_src\pkg_logic.cpp:289-304`

历史遍历列表最高只到 Bioconductor 3.22。Bioconductor 3.23 已于 2026-04-29 发布，因此在版本推断不适用或推断失败时，当前遍历不会覆盖 3.23。

`InferBiocVersion` 还是面向特定版本规律的启发式公式，却应用于所有 Bioconductor 包。特别是主版本 1、次版本 34 至 48 的分支返回 0 至 7，最终构造为 3.0 至 3.7，映射存在明显偏移。虽然之后会回退遍历，但会增加无效请求和等待时间。

### 8. HTML/JSON 解析失败可能被当作命中，并阻断后续回退

**影响等级：中**

**位置：**

- `cpp_src\app_window.cpp:184-200`
- `cpp_src\app_window.cpp:378-380`
- `cpp_src\app_window.cpp:459-463`
- `cpp_src\app_window.cpp:483-545`

Bioconductor 页面只要返回 HTTP 200 就立即设置 `anyFound = true`，即使版本字段解析为空也会投递“找到”结果。在快速模式下，这会阻断 GitHub 回退。

解析器依赖固定的 `<td>` 文本和手写 JSON 字段扫描。页面增加属性、字段转义或结构变化后，程序可能出现“HTTP 成功但业务解析失败”，当前状态模型无法识别这种情况。

### 9. 串行历史扫描可能表现为长时间无响应，但没有发现无限循环

**影响等级：中**

Bioconductor 历史版本最多串行尝试 23 个版本，每个请求配置 30 秒超时。单包理论等待可达到数分钟。普通全源失败还会依次尝试 CRAN、四类 Bioconductor、r-universe 和 GitHub。

`while (retrySearch)` 的大小写重试由 `hasRetriedCasing` 限制为一次，本次未发现确定性的无限循环。主要问题是串行请求数量大、缺少重试预算和阶段进度，用户可能将长等待误认为检索卡死。

### 10. 次要逻辑问题

1. `ConfigSnapshot.cranMirror` 已保存，但检索 CRAN 时固定使用 `https://cloud.r-project.org`，配置镜像只影响生成的 R 命令。
2. GitHub 只特殊处理 403，没有统一处理 401、404、422、429 和 5xx。
3. `std::thread` 构造发生异常时，`m_searchActive` 已经置为真，且窗口过程没有异常边界。
4. 窗口销毁后，队列中尚未处理的 `WM_SEARCH_LOG` 和 `WM_SEARCH_PROGRESS` 所携带堆对象可能无法释放。

## 实施方案

### 方案概述

优先重构任务取消和结果状态，不建议继续在现有“双线程共同关闭句柄”的模型上追加原子操作。推荐由工作线程保持 WinINet 句柄唯一所有权，UI 线程只设置取消标志；如必须即时取消同步网络调用，应改用文档支持的异步 WinINet 模式，或迁移到具有明确取消句柄模型的 WinHTTP。

### 详细步骤

| 步骤 | 操作 | 负责模块 | 风险等级 |
|------|------|----------|----------|
| 1 | 建立任务状态枚举：Running、Succeeded、Cancelled、NetworkFailed、InternalFailed | `app_window.hpp` | 低 |
| 2 | 将根会话和请求句柄改为工作线程唯一所有权，删除 UI 线程直接关闭同步句柄的逻辑 | `app_window.cpp` | 高 |
| 3 | 使用统一退出守卫负责清理句柄、清空原子状态和投递终态消息 | `app_window.cpp` | 中 |
| 4 | 区分 NotFound、TransportError、HttpError、ParseError、RateLimited | `app_window.hpp/cpp` | 中 |
| 5 | 取消后禁止显示“全量完成”，禁止自动生成基于部分结果的最终命令 | `app_window.cpp` | 中 |
| 6 | 将 Bioconductor 版本来源改为可维护数据或动态获取，修正启发式推断边界 | `pkg_logic.cpp` | 中 |
| 7 | 为 HTML/JSON 解析增加明确成功条件和失败日志 | `app_window.cpp` | 中 |
| 8 | 增加离线状态机测试、网络错误注入测试和停止竞态压力测试 | `报告\ai_codes\` | 中 |

### 预期效果

1. 停止操作不会跨线程重复关闭同步 WinINet 句柄。
2. 用户可以明确区分完成、取消、网络失败和包不存在。
3. 部分结果不会被包装成完整检索结果。
4. 网络临时故障不会批量制造“包不存在”的假阴性。
5. 新版 Bioconductor 发布后不会因硬编码列表过期而漏检。

## 验证结果

使用 MinGW g++ 以 C++17、`-O2 -Wall -Wextra -Wpedantic -Wshadow` 编译当前源码，构建成功。编译器报告的主要告警为函数指针转换、未使用参数、变量遮蔽和误导性缩进，未出现编译错误。

本次未执行真实 GUI 停止竞态压力测试。句柄问题依据代码控制流和微软官方 WinINet 使用约束确认，不依赖运行时偶现结果。

## 最终建议

当前版本存在会造成检索中断、错误完成状态和批量假阴性的确定性逻辑问题，不能认为检索链路已经稳定。

修复优先级建议：

1. **P0：修复 WinINet 句柄唯一所有权和停止竞态。**
2. **P0：为取消、网络失败和正常完成建立不同终态。**
3. **P1：禁止把网络错误归类为包不存在。**
4. **P1：补齐异常清理和 InternetOpenW 失败通知。**
5. **P2：更新 Bioconductor 版本策略和解析器。**
