# 交互逻辑回退修复报告 - 202604101511

## 1. 故障描述
用户手动点击单选按钮（如 `remotes::install_url`）时，发现生成的脚本内容并未切换到对应的安装方法。体感表现为“按钮功能丢失”。

## 2. 根因分析
在解决“全量体检后自动生成版本化脚本”的需求时，我引入了一段“智能增强”代码：
```cpp
// 只要检测到版本号，就强制切换到 Version 模式
if (!targetVer.empty() && (effectiveMethod == InstallMethod::Auto || effectiveMethod == InstallMethod::Remotes)) {
    effectiveMethod = InstallMethod::Version;
}
```
问题在于该段逻辑直接在 `effectiveMethod` 上操作，且没有区分用户的原始意图。这导致即使运行在手动模式下，只要输入框有版本号，它就会强行覆盖用户的选择。

## 3. 核心修复
我将该逻辑的触发范围限制在了只有用户选择“Auto”模式时才生效：
```cpp
// 增加 m_currentMethod == InstallMethod::Auto 的判断
if (m_currentMethod == InstallMethod::Auto && !targetVer.empty() && ...) {
    effectiveMethod = InstallMethod::Version;
}
```
这样，如果用户主动点击了其它的 Radio Button，`m_currentMethod` 就不再是 `Auto`，程序会完全尊重用户的手动选择。

## 4. 验证与结论
已成功重新编译 `RLinkInstaller_PRO.exe`。通过手动点击各个选项，确认生成的指令能够跟随单选按钮的点击而实时变化，逻辑冲突已解除。
