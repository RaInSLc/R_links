## 问题分析

### 现状

当前 `cpp_src` 已经加入 BiocGit 分支的部分实现：`app_window.cpp` 在 Bioconductor release 包版本与用户指定版本不一致时遍历历史 Bioconductor 版本，并在命中时以 `source = 3` 写入 `SearchProgressInfo`；`UpdateGeneratedCommand()` 将 `source = 3` 路由到 `InstallMethod::BiocGit`；`pkg_logic.cpp` 生成 `remotes::install_git()` 命令并将 `3.18` 这类版本号转为 `RELEASE_3_18`。

但是，该实现仍处于补丁式接入状态。线程生命周期、解析边界、版本映射、UI 状态管理、持久化路径与接口一致性等问题没有形成统一设计。既有审计报告中的部分问题已经被局部缓解，例如历史 Bioc 页面解析增加了 `hend_td > hnext_td + 4` 和 `hwl > 0` 判断；但 CRAN、release Bioc、r-universe、GitHub fallback 等路径仍保留相同类别的边界缺陷。

### 根因

1. 搜索逻辑、命令生成逻辑、UI 状态与持久化逻辑集中在 `app_window.cpp`，缺少可复用的网络解析、版本映射和任务生命周期抽象。
2. BiocGit 功能按“在现有遍历流程上追加 source=3”的方式接入，没有把 Bioconductor 版本推断、历史版本枚举、命令生成约束设计为稳定接口。
3. Win32 UI 控件 ID、`InstallMethod` 枚举值和页面显示逻辑存在隐式连续编号假设，导致小范围改动即可破坏 UI 状态。
4. 网络请求使用同步 WinInet 调用并放入 detached 线程，停止、退出、超时与异常处理没有形成完整状态机。

2. **BiocGit 版本推断函数没有落地。** `2026-06-03_分析_BiocGit实现逻辑补全.md` 要求添加 `ParseMinorVersion()` 与 `InferBiocVersion()`，当前 `pkg_logic.cpp` 没有这些函数，`pkg_logic.hpp:51` 还保留了未实现的 `ParseInputLine()` 声明。结果是代码仍依赖 21 次历史 HTTP 逐一试探，效率和可维护性都没有达到报告目标。

3. **指定 Bioconductor 版本未命中时会生成语义不一致的安装脚本。** 当用户输入特定版本且历史遍历未命中时，`app_window.cpp:345-351` 回退为 release Bioc 结果；随后 `app_window.cpp:691-697` 路由到 `BiocManager::install()`。`pkg_logic.cpp:64-67` 的 BiocManager 命令无法指定包版本，但条件包装仍可能检查 `packageVersion(pkg) != "用户指定版本"`，形成“判断指定版本，安装 release 版本”的不一致脚本。

4. **解析防御只修补了历史 Bioc 分支，其他路径仍有崩溃隐患。** `app_window.cpp:241-249`、`276-286`、`374-381`、`432-434`、`451-461` 仍直接使用 HTML/JSON 字符串截取和 `MultiByteToWideChar()` 返回值分配缓冲区，没有统一校验 `wl > 0`、`end > start` 或空白字符串。该问题属于稳定性缺陷，不应在没有明确利用路径时描述为安全漏洞。

6. **配置读写与搜索线程存在数据竞争。** 搜索线程在 `app_window.cpp:173-176`、`194`、`412` 读取 `m_proxy` 和 `m_githubToken`，主线程在 `app_window.cpp:843` 写入这些成员。当前只有 `m_searchActive`、`m_stopRequested` 是 atomic，字符串成员没有锁，也没有在线程启动时复制配置快照。

7. **停止按钮不是可中断取消。** `app_window.cpp:777` 只设置 `m_stopRequested = true`，搜索线程仅在包循环顶部 `app_window.cpp:182` 检查该标记。当前请求如果阻塞在 `InternetOpenUrlW()` 或 `InternetReadFile()`，用户点击停止不会立即生效，也无法取消当前网络句柄。
