## 背景

在当前版本的 R 语言包安装命令生成器 C++ 源码中，存在若干项涉及多线程数据竞争、异常处理越界导致的崩溃隐患，以及路径管理和缓冲区安全性问题。为了提升软件在高负载及复杂网络环境下的鲁棒性和安全性，本报告针对 `cpp_src` 目录下的核心源文件（`pkg_logic.cpp`、`app_window.cpp` 等）进行了深入的逻辑审计，并提出了针对性的优化方案。

## 问题分析

### 现状

现有的 C++ 应用程序基于 Win32 API 编写，集成了网络扫描功能、UI 事件循环和多源安装命令的自动生成逻辑。为了提供实时的扫描进度，程序采用 `std::thread` 异步执行网络检索，并通过 `PostMessageW` 将状态传回主窗口线程。然而，当前在跨线程数据读写、HTML 文本解析、动态内存分配转换以及配置文件和历史记录文件定位等方面，存在明显的稳定性及安全隐患。

### 根因

1. **跨线程并发访问无保护（Data Race）**：工作线程 `SearchWorkerThread` 频繁读取 `pApp->m_proxy` 和 `pApp->m_githubToken`。在扫描期间，如果用户通过设置界面修改配置并调用 `SaveConfig`，主线程会改写这两个 `std::wstring` 变量。由于 `std::wstring` 不是线程安全的，并发读写导致未定义行为，特别是在主线程重新分配内存时，工作线程可能发生内存访问冲突。
2. **生命周期不一致与 Use-After-Free 隐患**：扫描线程以 detached 方式运行。如果用户在扫描未完成时直接关闭应用程序，主线程销毁窗口及 `AppWindow` 实例，但后台工作线程可能仍然在执行，并且会继续访问已被释放的 `this` 对象的成员变量，导致程序关闭时在后台崩溃。
3. **HTML 解析截断越界（`std::out_of_range`）**：在提取 CRAN 与 Bioconductor HTML 数据中的包版本时，使用 `html.substr(next_td + 4, end_td - (next_td + 4))`。由于没有验证 `end_td > next_td + 4`，若网络流截断或 HTML 结构异常，无符号数减法下溢会导致截取长度成为超大数，进而抛出 `std::out_of_range` 异常。由于工作线程没有捕获异常，会导致软件瞬间闪退。
4. **字符编码转换越界**：如果 `MultiByteToWideChar` 返回 `0`（非法 UTF-8 数据），程序执行 `new wchar_t[0]` 随后调用 `ver = vwb`。解引用空长度 of 的 `wchar_t` 数组指针是未定义行为，会触发 Access Violation 崩溃。
5. **历史记录保存使用相对路径**：`history.txt` 文件的读写使用了相对路径。如果程序工作目录（CWD）改变，历史记录将被写入错误位置甚至由于写权限不足而失败。
6. **不安全的缓冲区复制（wcscpy 溢出）**：`SaveConfig` 中对镜像 URL 兜底处理时，使用 `wcscpy(b3, ...)` 拷贝至固定大小 512 的字符数组中。如果 URL 长度超过限制，将发生栈缓冲区溢出。
7. **缺失 DPI 动态调整响应**：设置了 `PROCESS_PER_MONITOR_DPI_AWARE` 却未处理 `WM_DPICHANGED` 消息，导致跨显示器时界面失真模糊。

## 核心发现

1. **工作线程多线程并发安全漏洞**：`m_proxy` 和 `m_githubToken` 成员变量在主线程和 detached 工作线程之间并发读写，在没有加锁或使用只读副本的情况下，存在高概率的数据竞争。
2. **网络层异常逻辑处理缺陷**：在解析 HTML 及 API 返回的 JSON 数据时，对于空悬指针或异常的截断字符串没有做防御性长度检查，非常容易被异常网络响应触发导致崩溃。
3. **资源生命周期管理缺陷**：Detached 线程缺乏退出的同步机制。当窗口销毁时，工作线程仍在后台读取已经销毁的对象，属于经典 Use-After-Free 隐患。
4. **不安全 C 库函数的使用**：`wcscpy` 在未验证源字符串长度的情况下容易导致缓冲区溢出崩溃。

## 实施方案

### 方案概述

针对审计发现的七大问题，进行防御性代码重构：使用互斥锁（`std::mutex`）或线程安全的配置拷贝保护跨线程数据；增加 `std::wstring` 解析的防御性范围判断；在 MultiByteToWideChar 时确保返回值大于 0；引入 `std::wstring` 避免不安全的 C 风格拷贝函数；将历史记录文件路径替换为绝对路径；捕获并处理工作线程可能抛出的所有异常。

### 预期效果

重构后，软件将消除所有已知的多线程数据竞争未定义行为，完全避免了由于网页结构变动/网络截断导致的闪退。同时，历史记录的保存位置将锁定在二进制文件同级目录下，不受工作路径变化的影响，安全性与鲁棒性将得到极大的提升。

## 最终建议

建议立即对 `app_window.cpp` 的 detached 扫描线程逻辑进行以下重构：
