# SpaCET 匹配精度修复与 Chrome 搜索功能实施方案

> 生成日期: 2026-06-06 | 类型: 方案 | 状态: 草稿

## 背景

在 R 包依赖安装管理工具中，当全量检索 GitHub 源时，由于只根据 star 排序遍历候选，无法区分精确匹配与部分匹配，导致如 `SpaCET` 包在匹配时，错误地选择了 Star 数更高但包名并不完全相等的 `edzer/spacetime` 库。此外，用户需要一个一键快速在 Chrome 浏览器中搜索候选包名的功能，以便手动对疑难包进行检索验证。

## 问题分析

### 现状
1. R 包 `SpaCET` 进行体检时，GitHub 搜索 API 找到了 `edzer/spacetime` 和 `data2intelligence/SpaCET` 等多个包含 `spacet` 的候选仓库。在匹配逻辑中，它们都具有相同的优先级（GitHub 源，优先级为3）。导致最终总是选择了第一个被扫描到的 `edzer/spacetime`（因为该包 Star 数最多，排在候选结果最前列）。
2. 目前的 GUI 界面上没有提供直接在外部浏览器（如 Chrome）对包名进行二次查询的入口。

### 根因
1. 在 `app_window.cpp` 的 `UpdateGeneratedCommand` 逻辑中，当多个匹配结果来源相同（例如都是 GitHub）时，仅根据遍历顺序将首个匹配设为最佳匹配，未考虑 repo 名与包名是否“完全相等”这一更高权重。
2. GUI 功能设计上缺失快速跳转搜索引擎的联动机制。

## 核心发现

1. `SpaCET` 包的完全相等仓库是 `data2intelligence/SpaCET`，而 `edzer/spacetime` 的仓库名是 `spacetime`，只因包含部分字母而被误判为同等权重的候选。
2. 只要在优先级比对逻辑中引入包名与仓库名完全相等的匹配判定，即可完美纠正该偏差。
3. Windows 平台下可使用 `ShellExecuteW` 结合 `chrome.exe` 和常规参数，实现稳定的一键打开标签页并搜索。

## 实施方案

### 方案概述
1. 在 `resources\resource.h` 中为新按钮分配 ID：`IDC_BTN_CHROME_SEARCH` 为 `222`。
2. 在 `app_window.hpp` 中引入 `<shellapi.h>`。
3. 在 `app_window.cpp` 的 `CreateControls` 中，于“停止”按钮右侧添加“Chrome 搜索”按钮。
4. 在 `OnCommand` 处理函数中，针对 `IDC_BTN_CHROME_SEARCH` 按钮编写事件响应：获取输入包名、去重，并依次启动默认浏览器（优先使用 `chrome.exe`）搜索包名和 `R package + 包名`。
5. 在 `UpdateGeneratedCommand` 中优化 `bestRes` 的比对逻辑：同源优先级下，若候选仓库名与包名完全相等（忽略大小写），则优先选择。
