# mod_UI 工程加固方案

> 生成日期: 2026-06-08 | 类型: 方案 | 状态: 已确认

## 背景

`mod_UI\` 是独立的 Tauri 2 + React + Rust 桌面项目，用于生成 R 包安装命令并执行多源检索。用户要求进一步加强到高可靠状态，并考虑启动、配置、权限、输入、网络、持久化、构建和维护等影响因素。

## 问题分析

### 现状

1. README 启动说明过短，不能覆盖 Windows 环境、Rust PATH、构建产物和常见问题。
2. Tauri 生产 CSP 原为空，前端申请了较宽的 opener 插件权限。
3. 前端和后端缺少统一输入规模限制，批量搜索与检索任务在极端输入下缺少明确边界。
4. 配置、代理、Token、CRAN 镜像和 GitHub 仓库标识缺少后端标准化校验。
5. 配置或历史 JSON 损坏时会直接返回错误，影响应用可恢复性。
6. HTTP 响应读取缺少大小上限，外部页面异常响应可能造成不必要资源消耗。

### 根因

项目最初以功能迁移为主，优先完成旧功能覆盖和可构建性。加固阶段需要把权限边界、数据校验、失败恢复和操作说明提升为显式工程约束，并通过测试与构建命令验证。

## 核心发现

1. 前端没有必要直接拥有通用 URL 打开权限，浏览器搜索可以下沉到 Rust 命令并限制为固定搜索域名。
2. Tauri CSP 可以在不影响现有 React/Vite 构建的前提下收紧生产资源加载范围。
3. Rust 后端是可信边界，应承担输入规模、URL、代理、Token、GitHub 仓库和历史记录的最终校验。
4. 应用数据目录中的 JSON 配置需要具备损坏备份与默认回退能力，避免单个文件破坏导致启动失败。
5. README 必须给出可复制执行的 Windows 命令，尤其说明 `npm run dev` 与 `npm run tauri dev` 的区别。
