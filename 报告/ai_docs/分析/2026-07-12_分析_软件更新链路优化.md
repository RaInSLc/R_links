# 软件更新链路优化分析

> 生成日期: 2026-07-12 | 类型: 分析 | 状态: 草稿

## 背景

当前软件更新能力基于 Tauri v2 updater 插件接入，前端通过 `@tauri-apps/plugin-updater` 的 `check()` 与 `downloadAndInstall()` 执行检查、下载和安装。配置位于 `mod_UI\src-tauri\tauri.conf.json`，更新端点指向 GitHub Releases 的 `latest.json`。用户反馈“软件更新逻辑好像有问题”，因此本次重点检查前端更新交互、Tauri updater 配置、发布清单和构建发布脚本之间是否闭环。

## 问题分析

### 现状

项目已具备基础更新能力：`tauri-plugin-updater` 已写入 `Cargo.toml`，前端依赖已包含 `@tauri-apps/plugin-updater`，Tauri capability 已授权 `updater:default`，配置文件已设置 GitHub Release 端点和 minisign 公钥。本地打包脚本 `报告\build_exe.ps1` 明确说明仅生成本地便携版，不生成自动更新清单，正式发布依赖 GitHub Actions Release 工作流。

### 根因

当前主要缺口不在插件接入本身，而在发布和应用端体验之间缺少可验证闭环：仓库当前未看到 `.github\workflows\` 发布工作流文件，也未发现本地 `latest.json` 样例或生成脚本；前端只在设置页提供手动检查更新，安装完成后仅提示用户手动重启，没有应用内重启入口或状态持久化。只要 GitHub Release 中缺失 `latest.json`、清单签名不匹配、资产 URL 指向错误或版本号未大于当前版本，用户侧就会表现为“检查不到更新”或“检查更新失败”。

## 核心发现

1. **更新入口存在，但发布清单链路不可在仓库内复现**：`tauri.conf.json` 指向 `https://github.com/RaInSLc/R_links/releases/latest/download/latest.json`，但当前仓库未发现 `latest.json` 生成脚本、样例清单或 GitHub Actions 工作流文件。若发布流程依赖外部手工操作，后续版本很容易遗漏签名、URL 或版本字段。
2. **本地打包不会生成自动更新清单**：`报告\build_exe.ps1` 明确写明“仅用于生成本地便携版，不生成自动更新清单”。如果实际分发使用该脚本产物，用户端 updater 不会自动获得可用更新。
3. **前端更新完成后的闭环不足**：`checkForUpdates()` 调用 `downloadAndInstall()` 后仅设置“请手动关闭并重启应用”文本，没有提供重启按钮、重启确认、失败重试入口或“已下载待重启”状态。
4. **更新状态与诊断信息割裂**：设置页可以导出诊断，并会携带 `updateMessage`，但 updater 过程本身没有记录端点、当前版本、目标版本、失败阶段、错误类型等结构化信息，排查时只能依赖单条状态文案。
5. **当前版本管理必须严格同步**：`package.json`、`tauri.conf.json`、`Cargo.toml` 均为 `0.1.9`。Tauri updater 只会在远端版本高于当前版本时返回更新，若发布时任一版本字段未递增，会表现为“当前已是最新版本”。
6. **GitHub latest Release 语义可能引入误判**：端点使用 `/releases/latest/download/latest.json`，它依赖 GitHub 将目标 Release 标记为 latest。若存在草稿、预发布或补发旧版本，客户端可能读取不到预期清单。

## 实施方案

### 方案概述

建议优先补齐发布侧闭环，再优化应用内体验。发布侧确保每次正式版本都自动生成和校验 `latest.json`、签名和安装包资产；应用侧提供可重启、可重试、可诊断的更新状态。

### 详细步骤

| 步骤 | 操作 | 负责模块 | 风险等级 |
|------|------|----------|----------|
| 1 | 在仓库中补齐 Release 工作流或本地清单生成脚本，统一生成 `latest.json` | `.github\workflows\` 或 `报告\` | 中 |
| 2 | 在发布流程中校验 `latest.json` 的 version、platforms、signature、url 均指向真实资产 | Release CI | 低 |
| 3 | 将当前版本、远端版本、更新端点、失败阶段写入诊断导出 | `App.tsx` / `export_diagnostics` | 低 |
| 4 | 更新安装完成后提供“立即重启”按钮，调用 Tauri process relaunch 能力 | 前端设置页 / Tauri capability | 中 |
| 5 | 将“检查中、下载中、安装完成待重启、失败可重试”拆成明确状态枚举 | `App.tsx` | 低 |
| 6 | 增加更新检查按钮旁的当前版本展示，降低用户误判 | `SettingsView.tsx` | 低 |
| 7 | 保留 GitHub latest 端点，同时在文档中规定正式 Release 不能使用预发布承担自动更新 | 发布规范 | 低 |

### 预期效果

完成后，更新链路会从“插件可用但依赖手工发布正确性”提升为“发布产物、清单、签名、客户端状态和诊断信息均可追踪”。用户可明确知道没有更新、正在下载、安装完成待重启或具体失败原因。

## 风险评估

| 风险项 | 影响范围 | 可能性 | 缓解措施 |
|--------|----------|--------|----------|
| 引入立即重启按钮需要新增 Tauri 权限 | 设置页更新区域 | 中 | 只授权必要的 process relaunch 权限，并在按钮点击前二次确认 |
| 发布工作流调整可能影响正式发版 | Release 产物 | 中 | 先在测试 tag 上验证清单生成与校验，再用于正式 tag |
| latest 端点仍依赖 GitHub Release 状态 | 自动更新发现 | 低 | 发布规范中禁止将自动更新版本标记为 prerelease，并在 CI 中校验 release 状态 |

## 最终建议

最值得优先优化的是发布清单闭环，而不是前端按钮本身。若 `latest.json`、签名或资产 URL 不稳定，前端无论怎么优化都会表现不可靠。建议先补“清单生成与校验”，再补“安装完成后立即重启”和“诊断字段”。
