# mod_UI 代码健壮性复检修复方案

> 生成日期: 2026-06-24 | 类型: 方案 | 状态: 已确认

## 背景

在 mod_UI 后端和前端的复检中，发现了以下若干代码健壮性与工程质量问题：
1. 错误捕获和上报机制不健全。后端的 HTTP 异常（例如 5xx、网络中断、JSON 解析失败等）在底层的网络获取函数中抛出，但在高层调用处被简单丢弃，从而全部被错误地归类为“包未找到 (notFound)”，给用户造成误导。
2. 后端 Bioconductor 检索循环未完全遵守停止信号。
3. 前端部分关键组件（History 状态变迁、Search 日志过滤、设置按钮交互、UTF-8 字符边界）缺乏自动化单元测试覆盖。
4. CI 构建过程中，因并发锁定问题经常导致 LNK1104 错误。且三个主要版本控制文件容易出现版本分叉。

本方案旨在针对这些问题提供一套体系化的技术修复。

## 问题分析

### 现状

- `get_text` 和 `get_json` 的返回类型被设计为 `Result<Option<T>, String>`。
- 调用处（如 `search_cran` 中的 `get_text`、`search_github` 中的 `get_json`）要么产生编译错误（由于类型不匹配），要么使用 `if let Ok(Some(...))` 静默忽略了 `Err(String)`，使得当网络报错或代理不可达时，静默表现为 NotFound。
- `sanitizeStatus` 方法只允许 `found`、`notFound`、`timeout`、`rateLimited` 状态，其余一律转换为 `notFound`。
- CI/CD 在 Windows Runner 上频繁因为 `target` 目录共享锁导致构建终止。

### 根因

- 底层网络错误缺乏冒泡以及统一收集汇总机制。
- 状态清洗过滤器（`sanitizeStatus`）过度收缩了有效状态种类。

## 核心发现

1. **状态污染**：由于前端 `sanitizeStatus` 强行将非预期的状态转回 `"notFound"`，导致网络故障与“包确实不存在”的物理状态混淆。
2. **死锁/锁竞争**：在同一个 `target` 下的多任务并发构建在 Windows 环境下会导致二进制文件或 PDB 锁死。

## 实施方案

### 方案概述

- Rust 后端：重构搜索检索链路，所有检索源函数均返回 `Result<Option<SearchResult>, String>` 或 `Result<Vec<SearchResult>, String>` 并在 `search_one_package` 中做错误累加，当所有源全部失败且存在错误累加时返回 `"error"` 状态。
- 前端 TypeScript：在 status 列表中加入 `"error"`。渲染时若状态为 `"error"`，提示“检索异常”。
- 前端测试：编写 `useHistory.test.ts`、`useSearch.test.ts`、`SettingsView.test.tsx` 和 `App.test.tsx` 单元测试。
- CI/CD：指定 `CARGO_TARGET_DIR=target/ci_target`，并利用 Node.js 内联脚本对比版本。
