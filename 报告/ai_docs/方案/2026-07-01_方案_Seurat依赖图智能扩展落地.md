在 R 语言的生态中，包依赖网络极其错综复杂。特别是在单细胞测序分析领域，以 `Seurat` 为代表的生态包，其依赖链条庞大且深重，往往级联引入数十甚至上百个第三方包。

目前系统已具备基本的单包来源检索、缓存和脚本生成能力，但依然缺乏依赖链路的深度拓扑分析。用户在输入 `Seurat` 时，无法直观查看到它与其依赖包（如 `SeuratObject` 等）之间的具体有向关系、强弱关系和共享引用关系。

## 问题分析

### 现状

当前后端检索包主要通过获取 CRAN 或 Bioconductor 网页并从中提取最新版本，并不获取和解析包的 `DESCRIPTION` 描述文件。以 `Seurat` 为例：
1. 搜索 `Seurat` 仅能得到其最新版本（如 `5.1.0`）及 CRAN 来源。
2. `Seurat` 复杂的 Depends (如 `SeuratObject`)、Imports (如 `ggplot2`)、Suggests 等多达几十个的子包信息在检索中被忽略。
3. 前端接收到的 `SearchResult` 是完全扁平的结构，无法渲染依赖拓扑树，难以评估安装风险和依赖链路的冗余度。

### 根因

1. **模型定义局限**：原有的 `SearchResult` 仅针对单包的扁平检索设计，未承载任何依赖层级和强度的字段。
2. **缺少 DESCRIPTION 获取链路**：未建立获取 CRAN/Bioconductor/GitHub 包的 `DESCRIPTION` 纯文本机制，无法获得依赖元数据。
3. **缺少图结构拓扑算法**：后端缺乏依赖图的拓扑排序、BFS/DFS 遍历和去重机制，易在处理 Seurat 极其复杂的共享依赖与循环依赖时产生死循环或性能瓶颈。

## 核心发现

1. **Seurat 的依赖规模**：根据 CRAN 数据，Seurat 拥有 2 个 `Depends` 包、36 个 `Imports` 包、3 个 `LinkingTo` 包以及 48 个 `Suggests` 包。若不加限制地全量展开所有依赖包的二级/三级依赖，节点数将呈指数级增长，极易触发网络请求风暴。
2. **轻重依赖分级**：`Depends`、`Imports` 和 `LinkingTo` 是运行或编译的硬性条件，应被定义为 **重依赖** (Heavy Dependency)；而 `Suggests` 属于测试与可选功能，应被定义为 **轻依赖** (Light Dependency)。
3. **数据源最小成本获取**：可以直接请求 `https://cloud.r-project.org/web/packages/{package}/DESCRIPTION` 获取纯文本。由于其格式为 Debian control (RFC 822)，单包仅几 KB，比起下载几兆的 `PACKAGES.gz` 文件，解析单包的 `DESCRIPTION` 的网络与内存开销最低，适合动态按需拉取。
4. **共享依赖（Diamond Dependency）在 Seurat 中的体现**：Seurat 本身依赖 `Matrix` 和 `Rcpp`，其核心 Depends 包 `SeuratObject` 也同时依赖 `Matrix` 和 `Rcpp`。这构成了一个典型的菱形依赖结构。
5. **本地元数据缓存必不可少**：解析过的包依赖数据（如 `Seurat:5.1.0` 的依赖列表）需进行本地持久化缓存，以避免频繁检索中对常用基础包（如 `Rcpp`, `rlang`）的重复网络拉取。
