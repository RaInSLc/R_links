## 背景

现有 `cpp_src\` 项目采用 Win32 C++ 实现 R 包安装命令生成与多源包检索。为降低原生窗口代码的维护成本、改善界面交互并建立清晰的前后端边界，本轮在 `mod_UI\` 中新建独立的 Tauri 2 + React + Rust 项目。旧项目仅作为行为参考，未修改其源码、配置、历史文件或构建产物。

新项目的目标是覆盖旧项目的主要用户能力，包括安装脚本生成、CRAN/Bioconductor/GitHub 多源检索、历史版本匹配、快速与全量检索、停止任务、报告与日志、剪贴板、浏览器搜索、历史记录及网络配置。

## 问题分析

### 现状

1. 旧界面、网络请求、命令生成和持久化逻辑集中在 Win32 窗口类中，界面状态与业务逻辑耦合。
2. 搜索流程依赖 WinInet 句柄和窗口消息，新增来源或调整交互需要修改大范围原生代码。
3. 历史记录与网络设置直接写入旧程序目录，不适合独立应用的数据隔离。
4. 旧界面受控件布局与 DPI 适配限制，难以扩展检索结果表格、实时日志和响应式布局。

### 根因

旧项目采用单体式桌面架构，业务状态、网络生命周期和控件消息共用同一实现边界。该结构适合早期功能验证，但不利于后续持续扩展。重构需要将界面渲染、业务规则、网络访问和持久化拆分，同时保持安装命令和来源路由行为可追踪。

## 核心发现

1. `cpp_src\pkg_logic.cpp` 提供了输入解析、命令生成、条件安装、版本推断和批量检查脚本等核心规则，适合迁移为纯 Rust 模块并通过单元测试固定行为。
2. `cpp_src\app_window.cpp` 的搜索流程包含 CRAN、Bioconductor Release、Bioconductor 历史版本、r-universe、GitHub API 和 GitHub DESCRIPTION 验证，需要由 Rust 异步网络层统一承载。
3. Tauri 2 的命令调用、事件机制、剪贴板插件和打开器插件能够覆盖旧项目的窗口消息、剪贴板及浏览器调用能力。
4. 配置与历史记录应写入 Tauri 应用数据目录，避免与旧 `config.ini`、`history.txt` 共享状态。

- React/TypeScript：负责工作台、报告、历史、设置视图以及本地交互状态。

- Rust 业务模块：负责输入解析、命令生成、网络检索、来源优先级、配置及历史持久化。
