### 根因

当前剩余问题的根因不是单个函数错误，而是 demo 经过多轮功能叠加后，职责边界仍集中在 `app_window.cpp`：

1. UI 事件、网络请求、HTML/JSON 解析、包版本推断、搜索状态机和命令生成路由仍耦合在同一模块。
2. Win32 原生消息循环适合轻量本地工具，但手写异步取消、HTTP 超时、结果模型和控件状态时，维护成本快速上升。
3. Bioconductor、CRAN、GitHub、r-universe 都是外部变化源，继续用字符串扫描实现数据解析，会持续产生兼容性维护成本。

## 核心发现

1. **Bioconductor 版本推断公式仍不正确。** `pkg_logic.cpp:279-283` 返回 `3 + (minor - 34) / 2`，而 `app_window.cpp:347-349` 又将返回值转换为 `3.(biocVer - 3)`。以 `GSVA 1.50.x` 为例，`minor = 50` 时函数返回 `11`，最终推断为 `3.8`，而目标应为 `3.18`。该问题会导致优先推断路径失效，只能依赖历史遍历兜底。

2. **停止与关闭逻辑仍有句柄竞争风险。** `app_window.cpp:933-939` 和 `app_window.cpp:700-710` 会在 UI 线程关闭 `m_hInternet`，工作线程同时持有 `hInt` 并在完成时再次读取和关闭。当前没有互斥锁、原子交换或 RAII 包装，属于稳定性风险；建议用专门的请求上下文管理 WinInet 句柄。

3. **没有设置 WinInet 超时。** 搜索线程会在多个 `InternetOpenUrlW()` 与 `InternetReadFile()` 调用上阻塞。虽然停止按钮现在会关闭根句柄，但未显式设置连接、发送、接收超时，退出和停止体验仍可能受网络状态影响。

4. **异常路径可能泄漏句柄并吞掉诊断信息。** `SearchWorkerThread()` 外层 `catch (...)` 只吞掉异常，未记录错误，也未确保 `m_hInternet` 和当前请求句柄统一释放。建议将 `HINTERNET` 封装为 RAII 类型，并在异常路径发送明确日志。

5. **GitHub fallback 被“找到任一源即跳过”策略弱化。** `app_window.cpp:427` 在 CRAN 或 Bioconductor 命中后跳过 GitHub 查询。这提升了速度，但会削弱此前“多源包同时展示 CRAN/Bioc/GitHub 来源”的能力。建议改为用户可选的 `快速模式` 与 `全量模式`，避免功能语义隐式变化。

6. **大小写纠正使用 `goto` 重试流程，后续可读性不足。** `app_window.cpp:251` 定义 `RETRY_SEARCH` 标签，`app_window.cpp:467` 通过 `goto RETRY_SEARCH` 重跑搜索。该写法在当前小范围内可以工作，但随着搜索源继续增加，建议改为循环状态机或拆出 `SearchPackageOnce()`，避免重试路径破坏局部作用域。

7. **HTML 与 JSON 解析仍是字符串扫描。** `SafeExtractTDValue()` 只匹配字面量 `<td>`，遇到 `<td class=...>`、大小写变化、换行属性或 Bioconductor 页面结构调整时仍可能失效。r-universe 和 GitHub API 也仍使用手写 JSON 字段扫描。建议改用稳定数据源或解析库。

8. **历史记录路径转换使用系统 ANSI 代码页。** `app_window.cpp:893-903` 将 `std::wstring` 路径转为 `CP_ACP` 后交给 `std::wofstream`/`std::wifstream`。当前路径是 `Z:\R_links` 时可用，但如果部署路径包含非系统代码页字符，历史记录读写可能失败。建议使用 `_wfopen`、`std::filesystem::path` 或支持 wide path 的文件 API。

9. **配置中的 GitHub Token 仍以明文 INI 保存。** 对个人单机 demo 可以接受；如果分发给多人或部署在共享主机，应考虑 Windows DPAPI 加密保存。该项是产品化增强建议，不在当前证据下标记为安全漏洞。

10. **构建脚本不适合作为自动验证入口。** `cpp_src\build.bat` 会写入正式 exe 和 `resources\app.res`，并带 `pause`。当前评审只能手写非破坏性编译命令。建议增加一个 CI/验证专用构建入口，将产物写入隔离目录，避免覆盖正式产物。

11. **缺少离线逻辑测试。** 当前核心逻辑已经包含版本推断、命令生成、HTML/JSON 解析、输入解析、大小写重试和多源优先级，但没有离线测试样例。建议在 `报告\ai_codes\` 先建立轻量验证原型，再决定是否将测试纳入正式工程。

## 最终建议

### 关于继续增强

短期建议继续保留 C++ demo，但只做收敛型增强：修正 Bioc 推断公式、完善停止/超时、增加非破坏性构建入口、拆出搜索与解析逻辑。不要继续在 `app_window.cpp` 内追加新来源或新 UI 状态，否则维护成本会继续上升。

### 关于技术选型

如果目标只是“Windows 单机、免运行时、轻量分发”的 demo，C++/Win32 是可以接受的，优势是体积小、启动快、无需额外运行环境。

如果目标是长期可维护工具，C++/Win32 不是最优解。当前功能本质是网络查询、HTML/JSON 解析、版本规则、脚本生成和表单 UI，最消耗维护成本的部分不是性能，而是异步、解析、测试和可变外部数据源。更优路线如下：

1. **Windows 产品化优先：C#/.NET 8 + WPF 或 WinUI。** 适合继续做桌面工具，异步 HTTP、JSON、配置、日志、测试和安装包体系都更成熟。
2. **内部快速迭代优先：Python + PySide6。** 与 R/Python 数据工具链更接近，HTTP、JSON、HTML 解析和打包更轻松，适合内部 demo 快速演进。
3. **跨平台 Web 化优先：Tauri + TypeScript。** 适合未来需要更复杂 UI 或跨平台分发，但初期工程复杂度高于 Python/C#。

综合判断：当前 C++ 版本适合作为 proof-of-concept 和 Windows 轻量演示版；如果后续还要继续增加包源、缓存、批量任务、配置安全、日志导出和测试体系，建议优先转向 C#/.NET；如果主要是内部自用和快速验证，Python/PySide6 是更现实的低成本方案。
